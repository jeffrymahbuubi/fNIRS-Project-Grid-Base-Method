# Setting Up the Project

## Overview

This guide covers two things:

1. **Ruflo / Claude-Flow** — the local AI coordination layer used to work on this project
2. **Cross-platform hardware setup** — how to flash and run the RuView firmware on a XIAO ESP32-S3 from macOS, Linux, or Windows

---

## Part A — Ruflo Setup

### 1. Install Ruflo

Initialize a package file if one does not exist, then install Ruflo as a dev dependency:

```bash
npm init -y
npm install ruflo --save-dev
```

### 2. Add the Init Script

Add the following entry to the `scripts` section of `package.json`:

```json
{
  "scripts": {
    "init-flow": "ruflo init"
  }
}
```

### 3. Run the Initializer

```bash
npm run init-flow
```

This generates the following files and folders:

| Path | Description |
|------|-------------|
| `.claude/` | Claude Code configuration directory |
| `.claude-flow/` | Ruflo runtime directory |
| `.mcp.json` | MCP server configuration |

### 4. Update `.mcp.json`

The default `.mcp.json` references `@claude-flow/cli@latest`. Replace it with the Ruflo package and enable `autoStart`:

**Before:**

```json
{
  "mcpServers": {
    "claude-flow": {
      "command": "npx",
      "args": [
        "-y",
        "@claude-flow/cli@latest",
        "mcp",
        "start"
      ],
      "env": {
        "npm_config_update_notifier": "false",
        "CLAUDE_FLOW_MODE": "v3",
        "CLAUDE_FLOW_HOOKS_ENABLED": "true",
        "CLAUDE_FLOW_TOPOLOGY": "hierarchical-mesh",
        "CLAUDE_FLOW_MAX_AGENTS": "15",
        "CLAUDE_FLOW_MEMORY_BACKEND": "hybrid"
      },
      "autoStart": false
    }
  }
}
```

**After:**

```json
{
  "mcpServers": {
    "claude-flow": {
      "command": "./node_modules/.bin/ruflo",
      "args": [
        "mcp",
        "start"
      ],
      "env": {
        "npm_config_update_notifier": "false",
        "CLAUDE_FLOW_MODE": "v3",
        "CLAUDE_FLOW_HOOKS_ENABLED": "true",
        "CLAUDE_FLOW_TOPOLOGY": "hierarchical-mesh",
        "CLAUDE_FLOW_MAX_AGENTS": "15",
        "CLAUDE_FLOW_MEMORY_BACKEND": "hybrid"
      },
      "autoStart": true
    }
  }
}
```

The three changes are:
- `"command": "npx"` → `"command": "./node_modules/.bin/ruflo"` (uses local install, no network fetch)
- The `-y` and `ruflo@latest` args are removed — no longer needed
- `autoStart: false` → `autoStart: true`

### 5. Verify the Connection

Launch Claude Code and run `/mcp`. The `claude-flow` server should appear as connected.

## Environment Variables Reference

The `env` block in `.mcp.json` is passed directly to the ruflo process at startup. Below is a reference for every variable the server reads, sourced from the ruflo source (`mcp-generator.ts`, `types.ts`, `mcp-server.ts`).

### Currently Configured

| Variable | Default value | What it does |
|---|---|---|
| `npm_config_update_notifier` | `"false"` | Suppresses npm's "update available" noise on stderr. Standard npm flag, not ruflo-specific. |
| `CLAUDE_FLOW_MODE` | `"v3"` | Selects the runtime generation. `v3` is the current one; exists for backwards compatibility with v2. |
| `CLAUDE_FLOW_HOOKS_ENABLED` | `"true"` | Activates the hooks system (PreToolUse, PostToolUse, UserPromptSubmit, SessionStart, etc.). Set to `"false"` to run a bare MCP server with no hook routing. |
| `CLAUDE_FLOW_TOPOLOGY` | `"hierarchical-mesh"` | Default swarm topology when `swarm_init` is called without an explicit argument. |
| `CLAUDE_FLOW_MAX_AGENTS` | `"15"` | Soft cap on agents per swarm. |
| `CLAUDE_FLOW_MEMORY_BACKEND` | `"hybrid"` | Storage backend for `memory_store` / `memory_search`. |

### Valid Values for Tunable Parameters

**`CLAUDE_FLOW_TOPOLOGY`**

| Value | Description |
|---|---|
| `"mesh"` | Peer-to-peer, no central coordinator |
| `"hierarchical"` | Queen + workers, tight anti-drift |
| `"hierarchical-mesh"` | Queen + workers with mesh fallback (recommended) |
| `"adaptive"` | Switches topology dynamically based on load |

**`CLAUDE_FLOW_MEMORY_BACKEND`**

| Value | Description |
|---|---|
| `"memory"` | In-process RAM only — lost on restart |
| `"sqlite"` | File-backed SQLite — persists, no vector search |
| `"agentdb"` | AgentDB with HNSW vector search (DiskANN) |
| `"hybrid"` | SQLite + AgentDB + HNSW + ONNX embeddings (recommended) |

### Additional Variables (Optional)

These are read by the server but not generated by `ruflo init`. Add them to the `env` block as needed.

| Variable | What it does |
|---|---|
| `CLAUDE_FLOW_CONFIG` | Path to a custom config file, overrides `.claude-flow/config.json` auto-discovery. |
| `CLAUDE_FLOW_MCP_TRANSPORT` | Force transport type: `"stdio"` (default for Claude Code), `"http"`, or `"websocket"`. |
| `CLAUDE_FLOW_CWD` | Override the working directory ruflo uses for PID files and relative paths. |

## Day-to-Day Command Reference

Commands live in `.claude/commands/` and are invoked with `/command-name` in the Claude Code prompt. They are **explicit** — not auto-triggered. For routine questions just prompt naturally; use commands when you want a specific structured workflow.

> **Note:** The three pillars — Commands, Agents, Skills — work as follows:
> - **Commands** = slash-command shortcuts you invoke explicitly (`/sparc:coder`)
> - **Agents** = specialized roles spawned *by Claude* based on your prompt (you rarely call these directly)
> - **Skills** = multi-step workflow templates, also slash-invoked (`/github:pr-manager`)

---

### Session Start

| Command | When to use |
|---|---|
| `/automation:session-memory` | Restore context and memory from a previous session |
| `/claude-flow-help` | Quick reference of all available claude-flow commands |

### Exploring an Existing Open-Source Project

Use these when you first encounter a repo and want to understand it before touching code.

| Command | When to use |
|---|---|
| `/github:repo-analyze` | Get an overview of repo structure, languages, key files |
| `/sparc:researcher` | Deep-dive research into how a specific module or concept works |
| `/sparc:analyzer` | Analyze code quality, patterns, and potential issues |
| `/sparc:ask` | Ask a targeted question about the codebase without triggering a full workflow |

### Planning Work

| Command | When to use |

| `/sparc:spec-pseudocode` | Write a specification + pseudocode for a feature before implementing |

| Command | When to use |

### Typical Day-to-Day Flow for an Open-Source Project

```
1. Start session
   → /automation:session-memory          (restore prior context)

2. Pick up an issue
   → /github:swarm-issue #<issue-number>  (decompose + plan)
   → /sparc:architect                     (if design needed first)

3. Implement
   → /sparc:coder  or  /sparc:tdd

4. Review & test
   → /sparc:reviewer
   → /sparc:tester

5. Submit
   → /github:pr-manager                  (create/manage PR)
   → /github:pr-enhance                  (polish if needed)
```

For **simple tasks** (single-file edits, quick questions) — just prompt naturally. Commands add value for **multi-step, structured workflows**.
